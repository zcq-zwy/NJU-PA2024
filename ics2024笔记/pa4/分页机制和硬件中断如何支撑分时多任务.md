# 分时多任务的具体过程：分页机制和硬件中断如何支撑 `pal` 与 `hello` 分时运行

## 先给结论

要让 `pal` 和 `hello` 在 `Nanos-lite + AM + NEMU` 上分时运行，至少要有两套机制同时成立：

- 分页机制负责回答：**“当前这条指令访问的内存属于谁？”**
- 硬件中断负责回答：**“当前这个程序什么时候必须停下来，把 CPU 让给别人？”**

两者缺一不可：

- 没有分页，两个程序会在同一片地址空间里互相覆盖
- 没有时钟中断，当前程序如果不主动 `yield()`，另一个程序就拿不到 CPU

所以，分时多任务不是“只有调度器”就够了，而是：

- `loader + 页表` 先把每个进程变成独立的“地址空间 + 初始上下文”
- `时钟中断 + trap + schedule()` 再把 CPU 在这些上下文之间轮流切换

---

## 一个需要先说清的前提

当前仓库里，`nanos-lite/src/proc.c` 的 `init_proc()` 直接装入的是：

- `/bin/pal`
- `/bin/bird`
- `/bin/onscripter`
- `/bin/sysmon`

也就是说，这个版本默认常驻的前台程序里**没有直接把 `/bin/hello` 放进初始进程表**。

但题目问“`pal` 和 `hello` 如何分时运行”，本质上是在问：

- 一个重量级用户程序（如 `pal`）
- 一个简单用户程序（如 `hello`）

它们为什么可以在这套系统上被轮流执行。

如果把当前某个用户进程替换成 `/bin/hello`，或者在 `NTerm` 中 `execve("/bin/hello", ...)`，底层走的仍然是同一条链路，因此下面的分析是成立的。

---

## 第一步：先把用户程序装进“各自独立的地址空间”

系统启动后，`nanos-lite/src/main.c` 会依次做这些初始化：

- `init_mm()`
- `init_device()`
- `init_disk()`
- `init_irq()`
- `init_fs()`
- `init_proc()`
- 最后执行一次 `yield()`

这里最关键的是 `init_mm()`、`init_irq()` 和 `init_proc()`。

### 1. `init_mm()`：先把虚存机制准备好

`nanos-lite/src/mm.c` 的 `init_mm()` 会调用：

- `vme_init(pg_alloc, free_page)`

而 `abstract-machine/am/src/riscv/nemu/vme.c` 的 `vme_init()` 做了两件决定性的事：

- 建立内核地址空间 `kas`
- 调用 `set_satp(kas.ptr)`，让硬件之后按照页表做地址翻译

这一步的意义是：

- 以后 CPU 不再把虚拟地址直接当物理地址用
- 而是要通过 `satp` 指向的页表，把虚拟地址翻译成物理地址

### 2. `init_proc()`：为每个进程建立自己的 `PCB + AddrSpace + Context`

`nanos-lite/src/proc.c` 中，`init_proc()` 对每个用户程序都会调用：

- `context_uload(&pcb[i], "/bin/xxx", argv, NULL)`

真正把用户程序“变成可调度进程”的，是 `nanos-lite/src/loader.c` 里的 `context_uload()`：

- `protect(&pcb->as)`：给这个进程创建页目录
- `loader(...)`：把 ELF 各段映射到用户虚拟地址空间
- `ucontext(&pcb->as, kstack, entry)`：构造这个进程的初始上下文
- `build_user_stack(...)`：建立用户栈

所以，`pal` 不是“读进内存就算完了”，而是被构造成了：

- 一份自己的页表 `pcb->as.ptr`
- 一份自己的用户代码/数据/栈映射
- 一份自己的内核栈
- 一份未来可恢复的 `Context`

若 `hello` 被同样装入，它也会拥有自己独立的一整套这些状态。

---

## 第二步：分页机制为什么能让两个程序互不踩内存

### 1. `protect()` 先复制内核映射，再给用户空间留位置

`abstract-machine/am/src/riscv/nemu/vme.c` 里的 `protect()` 会：

- 为进程分配一页新的页目录
- 把内核页表 `kas` 整页复制进去
- 把这份新页目录挂到 `as->ptr`

这表示：

- 所有进程共享同一套内核映射
- 但各自拥有独立的用户态映射

这正是“进程切换时内核仍然能工作”的基础。

### 2. `map()` 决定“这个虚拟页到底映射到哪一页物理内存”

`abstract-machine/am/src/riscv/nemu/vme.c` 的 `map()` 用的是 RISC-V Sv32 两级页表：

- `vpn1` 选页目录项
- `vpn0` 选二级页表项
- 最后写入叶子 PTE

它还会给页表项写上：

- `PTE_V`
- `PTE_R/W/X`
- `PTE_A/D`
- 对用户页额外加 `PTE_U`

因此：

- `pal` 的某个虚拟地址，比如 `0x40001000`
- `hello` 的同一个虚拟地址，比如也叫 `0x40001000`

最终可以映射到**完全不同**的物理页。

“虚拟地址相同，但物理地址不同”，正是分页机制支撑多进程隔离的核心。

### 3. 真正做地址翻译的是 NEMU 里的 MMU

`nemu/src/isa/riscv32/system/mmu.c` 的 `isa_mmu_translate()` 会：

- 从 `cpu.satp` 取当前页表根
- 按 Sv32 查两级页表
- 检查 `PTE_V/R/W/X/U/A/D`
- 决定这次取指、读、写是否合法

所以，当前到底访问到 `pal` 的内存还是 `hello` 的内存，不是 C 代码“口头约定”的，而是 NEMU 在每一次访存时按照 `satp` 和页表**硬算出来**的。

### 4. 进程切换本质上也在切页表

`abstract-machine/am/src/riscv/nemu/vme.c` 里的：

- `__am_get_cur_as(c)` 会把当前 `satp` 记到 `c->pdir`
- `__am_switch(c)` 会把 `c->pdir` 再写回 `satp`

于是：

- 恢复 `pal` 的上下文前，先把 `satp` 切到 `pal` 的页表
- 恢复 `hello` 的上下文前，先把 `satp` 切到 `hello` 的页表

这就是“同一个 CPU，为什么一会儿看见 `pal` 的地址空间，一会儿看见 `hello` 的地址空间”的原因。

---

## 第三步：硬件中断怎么把正在运行的程序强行打断

分页只解决了“隔离”，还没解决“轮流执行”。

要想让 `pal` 跑一会儿，再让 `hello` 跑一会儿，必须有一个外力定期打断当前程序，这个外力就是**时钟中断**。

### 1. NEMU 的定时器先产生设备中断请求

`nemu/src/device/timer.c` 中：

- `init_timer()` 注册 `timer_intr()`
- `timer_intr()` 会调用 `dev_raise_intr()`

而 `nemu/src/device/intr.c` 的 `dev_raise_intr()` 只是做一件事：

- `cpu.INTR = true`

含义是：设备告诉 CPU，“有中断来了”。

### 2. CPU 执行流在指令边界检查中断

NEMU 的执行循环里会调用：

- `isa_query_intr()`

`nemu/src/isa/riscv32/system/intr.c` 中，如果：

- `cpu.INTR == true`
- 并且 `mstatus.MIE == 1`

就返回定时器中断号 `IRQ_TIMER`。

### 3. `isa_raise_intr()` 负责进入 trap

同一个文件里的 `isa_raise_intr()` 会：

- 关中断
- 把原来的 PC 写进 `mepc`
- 把中断原因写进 `mcause`
- 保存之前的特权级到 `mstatus`
- 把 `cpu.pc` 跳到 `mtvec`

而 `mtvec` 是谁设置的？是 `abstract-machine/am/src/riscv/nemu/cte.c` 里的 `cte_init()`：

- `csrw mtvec, __am_asm_trap`

所以时钟中断最终会跳进：

- `abstract-machine/am/src/riscv/nemu/trap.S::__am_asm_trap`

---

## 第四步：trap 入口如何把“正在跑的那个程序”冻结下来

`trap.S` 里的 `__am_asm_trap` 是整个抢占过程最关键的入口。

它做的事情可以概括成：

- 保存通用寄存器
- 保存 `mcause / mstatus / mepc`
- 组织成一个 `Context`
- 调 C 函数 `__am_irq_handle`

这一步之后，当前程序就不再只是“CPU 里的一团瞬时状态”了，而是被固化成了一份内存里的 `Context` 结构。

这份 `Context` 就是以后恢复执行所需要的全部现场。

---

## 第五步：从中断事件进入调度器

`abstract-machine/am/src/riscv/nemu/cte.c` 中的 `__am_irq_handle()` 会：

- `__am_get_cur_as(c)`：先把当前页表根记下来
- 根据 `c->mcause` 区分是 `EVENT_IRQ_TIMER`、`EVENT_YIELD` 还是 `EVENT_SYSCALL`
- 调用户注册的处理函数 `user_handler(ev, c)`

这个 `user_handler` 就是 Nanos-lite 在 `nanos-lite/src/irq.c` 里注册的 `do_event()`。

而 `do_event()` 对两类事件都会直接调度：

- `EVENT_IRQ_TIMER`
- `EVENT_YIELD`

代码上都是：

- `return schedule(c);`

也就是说，中断处理的终点不是“处理完再回原进程”，而是把当前 `Context` 交给调度器，让调度器决定**下一个要恢复谁**。

---

## 第六步：`schedule()` 如何从 `pal` 切到 `hello`

`nanos-lite/src/proc.c` 的 `schedule(Context *prev)` 可以分成两步理解。

### 1. 先保存当前进程的上下文

如果当前确实是一个用户 PCB，就执行：

- `current->cp = prev;`

这意味着：

- 刚才被 trap 保存下来的那份 `Context`
- 现在正式挂到当前进程 PCB 上

于是 `pal` 被中断时，它的执行点、寄存器、返回位置、页表信息，都保存在 `pal->cp` 里。

若当前进程是 `hello`，它也同样会把现场保存在 `hello->cp` 里。

### 2. 再选出下一个进程，并返回它的 `Context`

这个仓库里的策略是：

- 前台程序拿多个时间片
- `sysmon` 周期性穿插执行

虽然当前实现没有默认把 `hello` 放在常驻列表里，但如果把 `hello` 当成另一个用户进程，这一步本质上就是：

- 如果当前是 `pal`，下一次可能选中 `hello`
- 如果当前是 `hello`，下一次可能又切回 `pal`

最后 `schedule()` 返回：

- `return current->cp;`

也就是“下一个要运行进程”的上下文。

---

## 第七步：返回前为什么还能自动切到新进程的地址空间

`__am_irq_handle()` 拿到 `schedule()` 返回的新 `Context` 之后，会调用：

- `__am_switch(c)`

这个函数会把：

- `c->pdir`

重新写入：

- `satp`

于是：

- 如果选中的是 `pal`，就切到 `pal` 的页表
- 如果选中的是 `hello`，就切到 `hello` 的页表

最后回到 `trap.S`：

- 恢复寄存器
- 恢复 `mstatus`
- 恢复 `mepc`
- 执行 `mret`

从 CPU 视角看，就是：

- 刚才还在执行 `pal`
- 现在 `mret` 一下，突然继续执行 `hello`

其实它不是“突然”，而是前面已经做完了：

- 保存旧上下文
- 选择新上下文
- 切换新页表
- 恢复新寄存器

---

## 把整个链路串成一条时序

假设现在轮到 `pal` 在跑，接下来一个时钟 tick 到来：

1. `pal` 正在用户态执行
2. NEMU 定时器到期，`timer_intr()` 调 `dev_raise_intr()`
3. CPU 在指令边界发现 `INTR`，`isa_query_intr()` 返回 `IRQ_TIMER`
4. `isa_raise_intr()` 写 `mcause/mepc/mstatus`，跳到 `mtvec`
5. `__am_asm_trap` 保存寄存器，构造 `Context`
6. `__am_irq_handle()` 把它识别成 `EVENT_IRQ_TIMER`
7. `do_event()` 调 `schedule(c)`
8. `schedule()` 先把 `pal->cp = c`
9. 调度器决定这次该轮到 `hello`
10. 返回 `hello->cp`
11. `__am_switch()` 把 `satp` 切到 `hello` 的页表
12. `trap.S` 恢复 `hello` 的寄存器和 `mepc`
13. `mret`
14. CPU 从 `hello` 上次停下来的地方继续执行

这就完成了一次真正的“抢占式分时切换”。

---

## 为什么说分页和中断缺一不可

### 只有中断，没有分页，不行

即使调度器每 10ms 切一次：

- `pal` 和 `hello` 还是会共用一套地址空间
- 栈、堆、全局变量会互相覆盖

这只是“轮流破坏同一块内存”，不是多进程。

### 只有分页，没有中断，也不行

即使两个进程各自有独立页表：

- 当前进程如果一直不主动陷入内核
- 调度器就没有机会接管 CPU

这只解决了“隔离”，没有解决“公平轮换”。

---

## 这套机制到底支撑了什么

对 `pal` 这种复杂程序来说，分页机制保证：

- 它的代码段、数据段、堆、用户栈都在自己的地址空间里

对 `hello` 这种简单程序来说，分页机制同样保证：

- 它看到的是自己的虚拟地址布局，而不是 `pal` 的

对两者共同来说，时钟中断保证：

- CPU 不会永远卡在其中一个程序里
- 内核能周期性收回控制权
- 调度器能决定下一次该恢复谁

因此，**分页机制提供“多个进程都能安全存在”的前提，硬件中断提供“多个进程都能轮流前进”的机会**。

---

## 最后的总结

如果只用一句话概括这道题，可以写成：

> 在我们的系统里，`context_uload() + protect() + map()` 先为 `pal` 和 `hello` 建立各自独立的虚拟地址空间；运行时，NEMU 定时器周期性触发硬件中断，经 `isa_raise_intr() -> __am_asm_trap -> __am_irq_handle() -> do_event() -> schedule()` 保存当前进程上下文、选择下一个进程，并通过 `__am_switch()` 切换 `satp` 到目标页表，最后 `mret` 恢复目标进程执行。分页机制解决“隔离”，硬件中断解决“抢占”，两者一起支撑了分时多任务。`
