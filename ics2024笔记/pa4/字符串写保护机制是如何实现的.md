# 理解计算机系统：字符串写保护机制是如何实现的

## 题目中的程序

```c
int main() {
  char *p = "abc";
  p[0] = 'A';
  return 0;
}
```

在 Linux 上运行它，通常会得到：

- `Segmentation fault`

这个现象不是“运气不好刚好炸了”，而是现代计算机系统从编译到硬件执行都在合作，尽量让这次非法写入**稳定失败**。

---

## 先给结论

这段代码之所以触发段错误，是因为它走过了这样一条完整的保护链：

1. **程序语言层**：`p` 指向的是字符串字面量，不是栈上数组
2. **编译器层**：编译器把字符串字面量放进只读常量区
3. **链接器/ELF 层**：这个常量区被放入 `.rodata`，并落进一个只读 `LOAD` 段
4. **装载器/内核层**：进程启动时，这个段被映射为只读页
5. **硬件/MMU 层**：CPU 执行写操作时发现目标页不可写，触发页保护异常
6. **Linux 内核层**：内核把这个异常翻译成对当前进程的 `SIGSEGV`

所以，“段错误会发生”并不是某一层单独保证的，而是整条链路共同保证的。

---

## 第一层：从 C 程序本身看，`p` 并不是一个可写数组

很多初学者会把：

```c
char *p = "abc";
```

误以为等价于：

```c
char p[] = "abc";
```

但它们完全不是一回事。

### `char *p = "abc";` 的含义

它表示：

- 先存在一个字符串字面量 `"abc"`
- `p` 只是拿到这个字面量首地址

也就是说：

- `p` 自己在栈上
- 但它指向的数据不在栈上

### `char p[] = "abc";` 的含义

它表示：

- 在当前栈帧里拷贝一份数组
- `p[0] = 'A'` 改的是自己那份副本

因此，这道题的关键不是“写字符串”，而是：

- 你在尝试修改一个**通常被放在只读区里的常量对象**

从 C 语言标准角度看，对字符串字面量写入本身就是**未定义行为**；而 Linux 上最常见、最合理的实现，就是直接用页保护把它打死。

---

## 第二层：编译器没有帮你偷偷复制，而是直接生成“对常量地址写入”的指令

我在 Linux 中用：

```bash
gcc -g -O0 -o /tmp/rostr /tmp/rostr.c
objdump -d -Mintel /tmp/rostr
```

看到 `main` 的关键汇编是：

```asm
lea    rax,[rip+0xecc]        # 2004
mov    QWORD PTR [rbp-0x8],rax
mov    rax,QWORD PTR [rbp-0x8]
mov    BYTE PTR [rax],0x41
```

这几条指令的意思非常直接：

- 先把字符串 `"abc"` 的地址取出来
- 存到局部变量 `p`
- 最后执行 `mov BYTE PTR [rax],0x41`

也就是说，编译器做的不是：

- “把字符串复制到一个可写缓冲区，再改它”

而是：

- “拿到只读常量的地址，然后真的去写那个地址”

所以问题不在“汇编器误会了你的意思”，而是机器码确实要求 CPU 去写一块只读内存。

---

## 第三层：链接器把字符串字面量放进 `.rodata`

继续用：

```bash
readelf -S /tmp/rostr
objdump -s -j .rodata /tmp/rostr
```

可以看到两个关键证据。

### 1. `.rodata` 段没有 `W` 标志

`readelf -S` 里，`.rodata` 这一项是：

- `PROGBITS`
- Flags 只有 `A`
- 没有 `W`

这表示它会被装入内存，但不应被当作可写节。

### 2. `"abc"` 确实就在 `.rodata`

`objdump -s -j .rodata` 可以看到类似：

```text
Contents of section .rodata:
 2000 01000200 61626300
```

其中：

- `61 62 63 00`

就是：

- `'a' 'b' 'c' '\0'`

所以证据链已经很清楚：

- 编译器把字面量保留下来
- 链接器把它放进 `.rodata`

---

## 第四层：ELF 程序头把 `.rodata` 放进一个只读的 `LOAD` 段

程序真正如何被装载，不是看 section header，而是看 program header。

用：

```bash
readelf -l /tmp/rostr
```

可以看到：

- `.text` 落在一个 `R E` 的 `LOAD` 段
- `.rodata` 落在一个只有 `R` 的 `LOAD` 段
- `.data/.bss` 落在一个 `RW` 的 `LOAD` 段

也就是说，链接器已经在 ELF 里表达了一个非常明确的意图：

- 代码：可读可执行，不可写
- 只读常量：可读，不可写
- 普通数据：可读可写

这一步非常重要，因为后面的装载器和内核会据此建立内存映射权限。

---

## 第五层：进程启动时，内核把 `.rodata` 映射成只读页

ELF 可执行文件启动时，Linux 内核会根据 program header 建立虚拟内存映射。

因此：

- `.rodata` 所在的 `PT_LOAD` 段
- 会被映射成 `r--p`

为了验证这点，我写了一个带 `sleep()` 的变体，先打印 `p` 的地址，再暂停一下，然后再执行写入。此时查看：

```bash
cat /proc/<pid>/maps
```

可以看到主程序对应的一段映射类似：

```text
603313d3a000-603313d3b000 r--p ... /tmp/rostr_maps
```

而打印出的 `p` 恰好落在这段只读映射里。

这说明：

- 进程地址空间里确实存在这页内存
- 但页权限只有读，没有写

所以这不是“地址无效”导致的段错误，而是“地址有效，但没有写权限”导致的保护异常。

---

## 第六层：CPU 执行 `p[0] = 'A'` 时，硬件/MMU 会拒绝这次写入

从 CPU 视角看，`p[0] = 'A'` 就是一条普通的 store 指令。

它会经历下面这个过程：

1. CPU 取到 store 指令
2. 计算目标虚拟地址，也就是 `p` 指向的位置
3. MMU 根据当前进程页表找到对应页表项
4. 检查该页是否具备写权限
5. 发现目标页是只读映射
6. 触发页保护异常

在 x86-64 上，这通常体现为：

- 一次 page fault（写只读页）

然后 CPU 会把控制权交给内核异常处理路径。

换句话说，真正阻止这次写入的“最后一道闸门”不是 C 编译器，而是**硬件页保护**。

---

## 第七层：为什么最后表现成 `Segmentation fault`

硬件抛出异常之后，Linux 内核会检查：

- 是不是用户态访问
- 访问地址是否落在合法映射内
- 是“页不存在”还是“权限不够”

对这种“用户态写只读映射”的情况，内核通常会给当前进程发送：

- `SIGSEGV`

shell 最终就会显示：

- `Segmentation fault`

所以，“段错误”其实是 Linux 在用户态暴露出来的上层表现；更底层的真实原因是：

- 硬件发现了非法写入
- 内核决定这个进程不能继续执行

---

## 计算机系统是如何保证“段错误会发生”的

这道题真正要回答的，不只是“为什么会错”，还要回答：

**为什么系统能比较稳定地把它拦下来？**

答案是因为有一条层层加固的保护链。

### 1. 编译器和链接器先把“常量”与“可写数据”分开放

- 字符串字面量进 `.rodata`
- 普通可写全局变量进 `.data`

这样装载时就有机会赋予不同权限。

### 2. ELF 程序头显式描述段权限

- `.rodata` 对应的 `LOAD` 段是 `R`
- `.data` 对应的 `LOAD` 段是 `RW`

内核不是“猜”哪块内存该只读，而是直接按 ELF 元数据来做。

### 3. 内核按段权限建立页表映射

这一步把“文件里的只读段”变成“内存里的只读页”。

### 4. CPU 每次写内存都检查页权限

因此即使程序员写出了错误代码，也绕不过硬件这一关。

所以这套保护不是“建议”，而是强制执行的。

---

## 如何用工具证明这条链路

这道题适合用“分层证据”来证明，而不是只说一句“`.rodata` 不可写”。

### 1. 证明程序真的崩溃

```bash
gcc -g -O0 -o /tmp/rostr /tmp/rostr.c
/tmp/rostr
echo $?
```

可观察到：

- 进程被 `SIGSEGV` 终止
- 返回码通常是 `139`

### 2. 证明字符串字面量在 `.rodata`

```bash
readelf -S /tmp/rostr
objdump -s -j .rodata /tmp/rostr
```

看两点：

- `.rodata` 没有 `W`
- 其中确实包含 `abc`

### 3. 证明机器码真的去写了那个地址

```bash
objdump -d -Mintel /tmp/rostr
```

看 `main` 中的：

- `lea ...` 取字符串地址
- `mov BYTE PTR [rax],0x41` 执行写入

### 4. 证明运行时映射确实是只读

写一个带 `sleep()` 的版本，在写入前查看：

```bash
cat /proc/<pid>/maps
```

确认：

- `p` 落在主程序的 `r--p` 映射范围内

### 5. 可选工具

如果环境允许，还可以用：

- `gdb`：看 fault 发生在那条 store 指令
- `strace`：看进程最终因 `SIGSEGV` 退出

不过即使不用它们，前面四类证据已经足够把整条链路说明白。

## 这道题最容易混淆的两个点

### 1. “字符串字面量不可写”不是 C 语法错误

编译器通常不会拒绝：

```c
char *p = "abc";
```

但这不代表写它就是合法的。

这里的关键是：

- “能编译通过”
- 不等于
- “运行时可以安全写入”

### 2. `SIGSEGV` 不等于“地址完全不存在”

很多人把段错误理解成“野指针访问到了不存在的地址”。

这只是 `SIGSEGV` 的一种常见情况。

本题更准确的说法是：

- 地址是存在的
- 映射也是存在的
- 但权限不允许写

本质是**保护异常**，不是单纯“地址不存在”。

---

## 最后的总结

如果只用一句话概括这道题，可以写成：

> `char *p = "abc";` 中的 `p` 指向字符串字面量；编译器和链接器把该字面量放进 `.rodata`，并通过 ELF 程序头把它放入只读 `LOAD` 段；Linux 在装载程序时把这段映射为只读页；当 CPU 执行 `p[0] = 'A'` 时，MMU 发现目标页没有写权限，触发保护异常，内核因此向当前进程发送 `SIGSEGV`。这就是字符串写保护机制的完整实现链路。`
