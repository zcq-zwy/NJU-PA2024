# XV6 Lab Base 说明

## 1. 目标

这份说明的目标不是介绍 xv6 本身，而是明确当前仓库里这条 `xv6-rv32-port` 线，已经被整理成什么状态，以及后续应该如何把它当作 **“在当前 NEMU 上做 xv6 实验的基线”** 使用。

当前目标是：

- 以后就在这个 `ics2024` 仓库里继续做 xv6 的实验
- 不再把重点放在“单纯把 xv6 跑起来”
- 而是把当前状态固化成一个可长期演化的 **lab base**

---

## 2. 当前基线是什么

当前基线不是 MIT 官方 `xv6-riscv` 的原始 lab handout 仓库，而是：

- 使用 `third_party/xv6-rv32` 作为 xv6 源码
- 在当前 `ics2024` 的 `riscv32-NEMU` 上补齐运行所需的平台能力
- 让 xv6 能在这套 NEMU 上启动、交互、运行基础测试

可以把它理解成：

> 一个 **适合继续做实验的 xv6-on-NEMU 基线**

而不是：

> 一份和 MIT 官方实验仓库逐字逐行完全一致的基线

---

## 3. 当前已经具备的能力

截至当前状态，这条基线已经具备：

- xv6 能启动到 shell
- 能在宿主机终端中直接与 xv6 shell 交互
- 支持 xv6 所需的 UART / CLINT / PLIC / virtio-mmio block
- 支持 xv6 运行所需的 S 态、CSR、异常委派、`sret`
- 支持基本的 SV32 页表翻译
- 支持页故障异常从 NEMU 正确注入 xv6 trap 路径
- 支持 xv6 文件系统镜像 `fs.img`
- 支持基础 smoke test
- `stressfs` 已经可以运行
- `twochildren` 这种较重的 `fork/exit/wait` 用例已经可以单独跑通

这意味着：

> 这条线已经不是“只能演示启动”的状态，而是“可以继续做实验”的状态。

---

## 4. 当前仓库里的关键入口

### 4.1 源码位置

- `third_party/xv6-rv32`

### 4.2 NEMU 配置

- `nemu/configs/riscv32-xv6_defconfig`

这是 xv6 专用配置，已经裁掉了一些对 xv6 实验不必要的设备，避免把无关开销混进测试结果里。

### 4.3 运行脚本

- `run_xv6_nemu.sh`

用途：

- 编译 xv6
- 生成 `kernel.bin`
- 编译 NEMU
- 直接启动 xv6

### 4.4 基础验证脚本

- `run_xv6_smoke_test.sh`

用途：

- 自动启动 xv6
- 自动注入一组最基本用户命令
- 检查 shell、文件系统、基础命令是否仍然正常

---

## 5. 如何判断当前基线是“可用”的

至少满足下面三个层次：

### 5.1 能启动

看到：

```text
xv6 kernel is booting
init: starting sh
$
```

### 5.2 能交互

在 shell 中能执行：

```sh
echo hi
ls
cat README
```

### 5.3 能跑基础回归

执行：

```sh
./run_xv6_smoke_test.sh
```

看到：

```text
[smoke] passed
```

这说明：

- shell 基本正常
- 用户态程序可执行
- 文件系统镜像基本正常
- 终端输入输出链路正常

## 6. 它和官方 xv6 lab 基线的差异

这件事必须明确，否则后面做实验时会混淆。

当前基线与 MIT 官方常见 xv6 lab 基线的主要差异包括：

### 6.1 当前是 rv32，不是官方常见的 rv64

当前使用的是：

- `xv6-rv32`

而 MIT 近年的官方 lab 多数围绕：

- `xv6-riscv`（rv64）

因此后面做实验时，可能会遇到：

- 指针/整数宽度差异
- 地址空间布局差异
- 某些实验说明默认按 rv64 叙述

### 6.2 当前平台是 NEMU，不是 QEMU

所以：

- 某些官方实验默认依赖的设备/行为
- 某些时间特性
- 某些异常/中断细节

在这里未必和 QEMU 完全一致。

### 6.3 当前目标是“能做实验”，不是“完全复刻官方环境”

因此当前基线更强调：

- 可运行
- 可调试
- 可验证
- 可逐步演化

而不是追求一开始就和官方环境完全等价。

---

## 7. 后续做 11 个 lab 的建议策略

后续最重要的不是继续“乱补功能”，而是保持分支历史清晰。

建议策略如下。

### 7.1 保留当前分支作为 lab base

当前分支：

- `xv6-rv32-port`

它已经可以被视为当前阶段的 **lab base**。

如果后面你希望命名更清楚，可以把它重命名为：

- `xv6-lab-base`
或
- `xv6-rv32-lab-base`

但这不是必须马上做的事情。

### 7.2 每个 lab 单独开一个分支

建议后续按实验拆分分支，例如：

- `xv6-lab-01-util`
- `xv6-lab-02-syscall`
- `xv6-lab-03-pgtbl`
- `xv6-lab-04-traps`
- `xv6-lab-05-lazy`
- `xv6-lab-06-cow`
- `xv6-lab-07-thread`
- `xv6-lab-08-lock`
- `xv6-lab-09-fs`
- `xv6-lab-10-mmap`
- `xv6-lab-11-net`

这样做的好处：

- 每个实验的改动边界清晰
- 出问题时容易回退
- 实验报告或答辩时容易说明

### 7.3 每个实验分支至少保留两个验证入口

建议每个 lab 分支都至少能跑：

```sh
./run_xv6_nemu.sh
./run_xv6_smoke_test.sh
```

这样可以保证：

- 新实验没有把启动链弄坏
- 基本 shell / 文件系统 / 用户程序没退化

---

## 8. 当前阶段不建议做的事

为了避免把重点从“做 lab”变成“做模拟器”，当前阶段不建议：

- 继续无限追求把所有 `usertests` 一次性全部跑穿
- 为了极端性能继续大改 NEMU
- 现在就追求和 MIT/QEMU 环境一模一样

原因很简单：

当前更需要的是一个 **稳定、可演化、可分支做实验** 的基线，
而不是一个“永远还没补完”的底层平台工程。

---

## 9. 当前推荐工作流

建议以后都按下面的方式推进：

1. 从当前 lab base 分支切出一个 lab 分支
2. 只做当前 lab 所需的改动
3. 先跑 `run_xv6_nemu.sh`
4. 再跑 `run_xv6_smoke_test.sh`
5. 如果实验有自己的专项测试，再额外补
6. 每个 lab 单独提交，保持历史清晰

---

## 10. 当前状态的一句话定义

当前这条线最准确的定义是：

> 它已经不是“把 xv6 跑起来的尝试版”，而是“在当前 NEMU 上继续做 xv6 实验的基线版”。

后续的重点应该从“继续补平台”逐步转向“围绕 lab 目标做功能实现与验证”。
